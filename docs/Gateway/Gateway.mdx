# Gateway

The gateway is the routing service of Kozz Bot. Boundaries and modules connect to it through Socket.io, introduce themselves with a signed payload and then exchange events through the gateway.

By default the gateway listens on:

```text
ws://localhost:4521
```

It also exposes an Express API server on port `3878`.

## Responsibilities

The gateway is responsible for:

- authenticating boundaries and modules;
- tracking connected services;
- parsing chat messages into commands;
- routing commands to modules;
- forwarding replies, media, reactions and deletes to boundaries;
- managing message proxies;
- forwarding resource requests and responses;
- exposing gateway-level resources.

## Message Flow

When a boundary emits a `message` event:

1. the gateway enriches it with boundary metadata;
2. configured proxies receive the message first;
3. the gateway attempts to parse the message as a command;
4. if parsing succeeds, it finds a module with the same name as the command;
5. the command payload is emitted to that module.

For example, `!ai qual foi a festa que o Pedro foi?` is parsed as:

```json
{
  "module": "ai",
  "method": "qual",
  "query": "qual foi a festa que o Pedro foi?"
}
```

If the module does not implement the parsed method, `kozz-module-maker` can route it to a `fallback` method.

## Gateway Resources

Modules and boundaries can ask the gateway for built-in resources.

### `all_boundaries`

Returns every connected boundary.

```json
{
  "response": [
    {
      "id": "3fxwNQovfSt9YJsLAAAe",
      "boundary": {
        "name": "kozz-baileys",
        "platform": "Baileys",
        "OS": "linux",
        "listeners": []
      }
    }
  ]
}
```

### `all_modules`

Returns every connected module.

```json
{
  "response": [
    {
      "name": "ai",
      "handler": {
        "id": "Y-y40LFeMN-xI920AAAB",
        "methods": ["fallback", "image", "transcribe"],
        "listeners": []
      }
    }
  ]
}
```

### `all_proxies`

Returns the active message proxy rules.

## Authentication

Every boundary and module signs its introduction payload with `keys/privatekey.pem`. The gateway verifies it with `keys/publickey.pem`.

If a service cannot connect, check:

- the key files exist in the expected folders;
- the service points to the correct `GATEWAY_URL`;
- the gateway and service agree on the socket path if `SOCKET_PATH` is configured.
