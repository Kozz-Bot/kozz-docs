# Kozz Bot

Kozz Bot is a distributed chat automation framework written in TypeScript. It is built around small services that communicate through [Socket.io](https://socket.io), so chat integrations, command handlers, dashboards and automation modules can evolve independently.

The project is currently composed of three main runtime roles:

- **Gateway**: receives events from every service and routes commands, resources, proxied messages and replies.
- **Boundary**: connects Kozz to a chat platform. The current WhatsApp boundary is `kozz-baileys`.
- **Module**: defines behavior. Modules can expose commands, listen to proxied messages, ask resources from other services and send messages through boundaries.

## Repositories

The current workspace is split into these repositories:

- [`Kozz-Gateway`](https://github.com/Kozz-Bot/Kozz-Gateway): the Socket.io gateway and API server.
- [`kozz-baileys`](https://github.com/Kozz-Bot/kozz-baileys): the WhatsApp boundary powered by Baileys.
- [`Gramonta-Bot`](https://github.com/Kozz-Bot/Gramonta-Bot): a real bot implementation built with Kozz modules.
- [`kozz-docs`](https://github.com/Kozz-Bot/kozz-docs): this Docusaurus documentation site.

## Current Status

Kozz Bot is still experimental. It is useful for personal automation and bot experiments, but the public APIs and repository layout may still change.

The WhatsApp integration currently uses Baileys. Older references to `kozz-wa`, `Kozz-Whatsapp` or `Whatsapp-web.js` are legacy names and should be treated as historical unless a specific project still uses them.

## Runtime Flow

1. A boundary receives a chat event from a platform.
2. The boundary normalizes the event into a Kozz payload and emits it to the gateway.
3. The gateway forwards the message through configured proxies, then tries to parse it as a command.
4. If a matching module is connected, the gateway sends the command payload to that module.
5. The module can reply, send media, react to messages, delete messages, ask resources or call other APIs.

This design is more infrastructure-heavy than a single-process chatbot, but it keeps platform integration, routing and behavior isolated.
